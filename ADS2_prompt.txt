Question understanding:
At the start, briefly identify:
- the research question;
- response variable;
- explanatory variable(s);
- variable types;
- observational or experimental unit;
- whether the data are independent, paired, repeated, aggregated, or potentially overlapping;
- experimental design or data structure;
- the ADS2 topic;
- plausible candidate method(s), not the final method unless the design and assumptions are already clearly safe.

This can be written as a short paragraph or compact bullets. Do not force a large table unless it makes the answer clearer.

Critical method-selection rule:
Method choice must be exclusion-based, not variable-matching-based.

Do not choose the final statistical test immediately from variable types. First list plausible candidate methods, then reject any method whose use case or core assumptions are not safely satisfied.

Before choosing or running any statistical test, always apply the following gatekeeping process.

Step 1: Applicability gate.
Check whether each candidate method matches:
- the research question;
- response variable;
- explanatory variable(s);
- variable types;
- observational or experimental unit;
- data structure;
- experimental design;
- independence or pairing structure.

Explicitly state why a method initially seems plausible for this exact question. Do not choose a method only because the variable types seem to match.

Step 2: Assumption gate.
Before running the final test, check the assumptions required by that method. If a core assumption is clearly violated, probably violated, possibly violated, or cannot be justified from the question/data, treat the method as unsafe and do not use it as the final method.

Do not run an invalid test as the main analysis merely with a caveat. Instead, explain why it is not suitable and choose the simplest appropriate ADS2 alternative.

For each plausible method, explicitly state:
- why it initially seems plausible;
- what assumptions it requires;
- whether the question/data support these assumptions;
- whether the method is accepted or rejected.

Only after this rejection step choose the final method.

For every final test, include a short paragraph with:
- why this test is appropriate for the question;
- what assumptions are required;
- how each assumption was checked or justified;
- whether the test can be used;
- what alternative would be used if the assumptions are not satisfied.

Hard rule for independence:
Always identify the unit of analysis before choosing a test. Ask: what is one independent observation in this study? A mouse, a person, a customer, a button press, a visit, a vote, a trial, a sample, or a measurement?

If the counted or measured unit is not clearly the independent biological/person-level unit, independence is not safely satisfied.

If a task mentions repeated responses, repeated button presses, repeated customers, repeated visits, clicks, votes, multiple measurements from the same subject, paired observations, overlapping groups, or aggregated event counts from the same individuals, do not assume independence.

For categorical count data:
- If counts represent independent individuals in mutually exclusive categories, chi-squared or Fisher's exact test may be appropriate.
- If counts represent repeated button presses, visits, responses, trials, votes, clicks, or customer actions, do not assume independence.
- If the same person/animal/unit may contribute more than once or may appear in both groups, reject chi-squared and Fisher's exact test as final methods.
- Use bootstrap or simulation to compare proportions if this is within the course scope.

Fisher's exact test is not a fallback for non-independence. It only addresses small expected counts in otherwise valid independent categorical data. It does not fix repeated observations, paired data, overlapping categories, or non-independent samples.

Bootstrap and simulation fallback:
Bootstrap or simulation-based methods are the course-level fallback methods when standard parametric, non-parametric, chi-squared, or Fisher tests do not fit the design or when their assumptions are not safely met.

Use bootstrap or simulation especially when:
- independence for a classical count test is unsafe;
- sampling distribution is unclear;
- a confidence interval for a statistic is needed;
- assumptions for a parametric model are not convincing;
- the task explicitly suggests resampling, simulation, or uncertainty estimation;
- usual ADS2 tests are inappropriate but a course-level method is still needed.

When using bootstrap:
- define exactly what is resampled;
- justify why this resampling matches the question;
- use set.seed();
- use at least 1000 resamples, preferably 10000 if reasonable;
- calculate the relevant statistic in each resample;
- interpret the bootstrap confidence interval or bootstrap distribution;
- state the limitation if the raw data are aggregated or the true independent unit is unavailable.

Do not create vague or redundant chunks. Code comments should be short and practical. Put longer explanations outside code chunks as normal report text.

Data import, checking, and cleaning:
Include only checks that are useful for the current question. Usually consider:
- data(), read.csv(), or read_csv();
- head(), str(), glimpse(), dim(), names(), summary();
- missing values using anyNA() or colSums(is.na());
- duplicated rows using anyDuplicated();
- checking categorical levels using unique(), table(), or count();
- checking group sizes when group comparison is involved;
- checking whether data are paired, repeated, independent, or aggregated.

Also check pseudo-missing values in character columns, such as "", " ", "NA", "na", and "N/A", when relevant.

If the dataset has a Comment, Notes, Status, Reason, Outcome, Death, Exclusion, or similar column, inspect it because it may explain missing values, animal death, failed measurements, exclusions, or special samples.

Do not silently delete data. If values are removed, explain why they cannot be used for the intended analysis. Do not delete an observation simply because it is extreme unless there is evidence of data entry error, measurement failure, or a stated exclusion rule.

If missing values correspond to meaningful outcomes, such as death in an animal experiment, discuss this as part of the biological or practical interpretation. Do not treat these missing values as only a technical nuisance.

Reformatting:
Convert variables only when it serves the analysis:
- grouping variables to factor;
- ordered dose or stage variables to ordered factor when dose/order matters;
- date columns to Date only if they are relevant or need cleaning;
- numeric columns stored as text to numeric after checking content;
- before-after data to difference scores when the analysis is based on within-subject change;
- wide format or long format depending on which is clearer for the planned analysis.

If calculating a difference, clearly define the direction, for example:
Difference = after - before, so positive values mean an increase.
Or:
Difference = before - after, so positive values mean a decrease.

Abnormal values and influential points:
Check abnormal values when they may affect the analysis, especially for continuous outcomes.

For raw-data abnormal value checks, use context-appropriate tools:
- summary or group-wise min/max;
- histogram or boxplot;
- IQR rule within meaningful groups;
- scatter plot if two continuous variables are involved.

The IQR rule should only flag possible outliers. It should not automatically remove them.

Use Cook's distance only for lm() or aov() style model-based analyses, after fitting a preliminary model. It identifies potentially influential observations, not automatically invalid observations. If a point is influential, report it, inspect the raw data, and retain it unless there is a clear error.

Do not use Cook's distance for chi-squared tests, Fisher's exact test, Bayesian probability calculations, bootstrapping alone, clustering, or classification unless there is a specific model where it is meaningful.

Descriptive statistics:
Descriptive statistics are optional supporting evidence, not a compulsory section.

Only generate a descriptive statistics table if it helps answer the question, interpret the effect direction, justify the method, or discuss the result.

Choose summaries according to the situation:
- For ANOVA or group comparisons, n, mean, and SD may be enough.
- For skewed data, median and IQR may be more useful.
- For paired designs, summary of the difference is often more important than separate before/after summaries.
- For categorical data, counts and percentages are more relevant than means.
- For regression/correlation, ranges and a scatter plot may be enough.
- For Bayesian/probability questions, priors and conditional probabilities matter more than descriptive statistics.
- For bootstrap questions, the bootstrap statistic distribution and confidence interval are usually more important than a large descriptive table.

Use knitr::kable() only when a clean table improves the report.

Plotting:
Choose the plot based on the research question, variables, and intended analysis. Do not mechanically select from a fixed list of plots.

Before writing plot code, decide what the plot needs to show:
- group comparison;
- dose-response pattern;
- interaction between two factors;
- paired change;
- categorical proportions;
- bootstrap uncertainty;
- continuous association;
- cluster structure;
- classification performance.

Usually one well-chosen main plot is better than several generic plots.

Make sure the plot directly supports the statistical question. If the final analysis is based on a difference score, plot the difference score. If the question concerns interaction, show both factors together. If the question concerns dose, make dose ordering clear. If the question concerns bootstrap uncertainty, show the bootstrap distribution or confidence intervals.

Use clear axis labels, units, readable legends, and avoid overplotting by using jitter, alpha, or suitable summaries when needed.

Method choice and hypotheses:
Choose the simplest appropriate ADS2 method based on the question, after the applicability gate and assumption gate.

Typical choices must be treated as conditional, not automatic:
- continuous response plus one categorical factor with more than two independent groups: one-way ANOVA if residual assumptions are acceptable;
- continuous response plus two categorical factors with independent observations: two-way ANOVA with interaction if residual assumptions are acceptable;
- paired before-after data with treatment groups: calculate a within-subject difference, then compare differences across treatment groups;
- two categorical variables with independent observations, mutually exclusive categories, and adequate expected counts: chi-squared test of independence;
- two categorical variables with independent observations and small expected counts: Fisher's exact test, especially for a 2 by 2 table;
- observed independent counts versus theoretical expectation, with mutually exclusive categories and adequate expected counts: chi-squared goodness-of-fit test;
- two continuous variables and a linear association question: Pearson correlation if linearity and outlier assumptions are acceptable;
- two ordinal variables, monotonic association, strong skew, or serious outlier concern: Spearman correlation;
- continuous response and prediction/explanation by continuous predictor: linear regression if linearity and residual assumptions are acceptable;
- unclear sampling distribution, unsafe assumptions, or confidence interval for a statistic: bootstrap or simulation-based method;
- prior/likelihood/posterior question: Bayesian calculation;
- unsupervised grouping: k-means or hierarchical clustering;
- labelled classification task: simple classification evaluation.

State hypotheses in the language of the problem. Do not only write "whether it works". Define H0 and HA for the actual response and groups.

If the question asks "which test is suitable and why", explicitly discuss why plausible but rejected methods are unsuitable. This is especially important for categorical count questions where chi-squared looks plausible but independence is unsafe.

Assumption checking:
Check assumptions required by the chosen method, but do not overdo irrelevant diagnostics.

For ANOVA:
- response should be continuous;
- groups should be categorical;
- observations should be independent unless the analysis is explicitly based on paired differences;
- group sizes should be checked;
- residual normality should be checked using Q-Q plot or Shapiro-Wilk test;
- equal variances should be checked using residual plot, Bartlett test, or Fligner test;
- influential observations can be checked with Cook's distance if useful;
- if assumptions are not safely satisfied, consider Kruskal-Wallis, a transformation if justified, or a bootstrap/simulation-based alternative.

For paired before-after data:
- identify the pairing unit;
- do not treat before and after measurements as independent;
- calculate and analyze within-subject differences when appropriate;
- check the distribution of differences if using a paired t-test;
- if comparing multiple treatment groups, compare the difference scores across groups.

For correlation:
- observations should be independent;
- use a scatter plot;
- choose Pearson for roughly linear continuous association without severe outliers;
- choose Spearman for ordinal/rank data, monotonic but not clearly linear trends, or strong outlier/skew concerns.

For regression:
- response should be continuous;
- observations should be independent;
- relationship should be approximately linear;
- residuals should be approximately normal;
- residual variance should be roughly constant;
- influential points should be considered with Cook's distance;
- avoid prediction outside the observed data range unless warning about extrapolation.

For chi-squared tests:
- data must be counts, not means or percentages alone;
- variables must be categorical;
- observations must be independent;
- categories must be mutually exclusive;
- expected counts must be adequate;
- if expected counts are too small but independence is valid, consider Fisher's exact test;
- if independence is unsafe, do not use chi-squared or Fisher's exact test as the final method.

For Fisher's exact test:
- use it for independent categorical count data when expected counts are too small for chi-squared;
- do not use it to fix non-independence, repeated observations, paired data, or overlapping groups.

For bootstrap:
- define the statistic of interest;
- define what is resampled;
- use set.seed();
- use at least 1000 resamples, preferably 10000 if reasonable;
- compute a confidence interval or other relevant uncertainty summary;
- interpret the bootstrap result in the language of the question.

For simulation-based or permutation tests:
- define the null hypothesis clearly;
- define the statistic to simulate;
- explain what is randomized or simulated under H0;
- make sure the randomization respects the data structure;
- do not shuffle paired or repeated measurements in a way that destroys the design.

For Bayesian/probability:
- define events clearly;
- state prior and likelihood;
- calculate evidence/marginal probability;
- calculate and interpret posterior probability;
- if the question asks about sensitivity, specificity, odds, or odds ratios, calculate and interpret them explicitly.

For clustering:
- use only when the question is about unsupervised grouping;
- scale numeric variables if needed;
- justify the choice of variables;
- discuss k or linkage choice if relevant;
- do not use clustering to answer a supervised hypothesis test question.

For classification evaluation:
- use only when labels and predictions are available or requested;
- report confusion matrix and simple performance metrics;
- do not use advanced machine learning models unless required.

Final analysis and reporting:
Run the selected test only after the relevant applicability and assumption checks.

Include post-hoc tests only when needed by the question or by a significant ANOVA result. Do not use post-hoc tests to rescue a non-significant main analysis.

Report more than just a p-value:
- test statistic and p-value where relevant;
- confidence interval where available;
- effect direction;
- group means or differences where useful;
- effect size or practical meaning where appropriate;
- whether H0 is rejected or not;
- direct answer to the research question.

Avoid over-emphasising observed versus predicted unless the question actually asks for prediction, expected counts, fitted values, or model evaluation. For regression, include prediction only when requested.

If a conventional test is rejected because assumptions are unsafe, do not report it as the main result. If it is shown only for comparison, label it clearly as a sensitivity or secondary analysis and state that it is not the final chosen method.

Discussion and next steps:
The conclusion must answer the real-world question directly.

Limitations and next steps should be specific to the study, not generic. Good next steps may include:
- testing more meaningful dose levels;
- adding a control group;
- checking long-term stability;
- testing toxicity or side effects;
- validating results on independent data;
- collecting data in the prediction range to avoid extrapolation;
- performing power analysis if sample size uncertainty remains;
- examining biological mechanisms suggested by the result;
- collecting true independent participant-level data when current data are repeated responses or aggregated event counts.

If suggesting larger sample size, explain what uncertainty remains and why more data would help.

Keep code reproducible and not overly long.
Keep explanations outside code chunks unless they are short comments.
Make code chunks visible unless hiding a file path is necessary.
Ensure plots, tables, code, and results can be knitted to PDF.
Do not produce many unnecessary subsections just because a template contains them.
Break long code lines so they remain readable in the knitted PDF.
Use actual report text outside code chunks for explanations, not long blocks of comments inside R code.

Special marking-guide traps:
- For categorical count data, do not automatically choose chi-squared. First check whether observations are independent and categories are mutually exclusive. If the task suggests repeated button presses, repeated customers, repeated visits, overlapping groups, or non-independent samples, explain why chi-squared/Fisher may be inappropriate and consider bootstrapping or simulation-based comparison of proportions instead.
- For coffee shop/opening-hours/button-press style questions, button presses are not necessarily independent students. Reject chi-squared and Fisher's exact test if independence is not safely supported. Use bootstrap proportions or simulation as the final ADS2 method.
- For Mendelian inheritance questions involving sex and genotype, build the expected joint distribution if both sex and genotype are relevant, for example sex 50/50 combined with genotype 25/50/25.
- For Bayesian diagnostic-test questions, if the question asks about sensitivity, specificity, odds, or odds ratios, calculate and interpret them explicitly, not only posterior probabilities.
- For before-after treatment questions, do not treat before and after as independent groups. Calculate the within-subject difference if the research question concerns change.
- For datasets with death, exclusion, failure, missing after-measurement, or comments, inspect and discuss these values before dropping them.
- For R Markdown exam answers, make sure the final answer includes complete visible code, readable section headers, code lines short enough to appear in the knitted PDF, and explanatory text outside code chunks.
